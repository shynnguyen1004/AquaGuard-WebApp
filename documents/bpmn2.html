Level 1 Overview Description

  • Entry Point (Process 1): Citizens and Rescuers can register or log in. Admin can only log in (Admin role is assigned by another Admin, not self-registered).
  • Citizen Can submit SOS (P2) during emergencies and track family on the map (P6). After submitting, the citizen waits until rescued.
  • Rescuer Can form/join a team (P5) and handle rescue missions (P3) individually or as a group.
  • Admin Can manage flood zones (P4) and oversee missions (P3) — assign rescuers or force-complete.
  • Cross-process links: SOS (P2) feeds into Rescue Queue (P3). Flood data (P4) appears on the live map for P6 and P3. Admin can intervene in P3 directly.

Registration & Login

The process of authenticating users and granting role-based access (Citizen — Rescuer — Admin)
flowchart TB subgraph Pool1["PROCESS 1: REGISTER AND LOGIN"] direction TB subgraph L1_User["USER"] S1((Start)) --> Visit["Visit AquaGuard website"] Visit --> AuthChoice{"Choose login method"} AuthChoice -- "Phone number" --> PhoneChoice{"Login or\nRegister?"} PhoneChoice -- "Login" --> InputLogin["Enter phone number\nand password"] PhoneChoice -- "Register" --> InputReg["Enter full name,\nphone number, password,\ngender, date of birth"] InputReg --> PickRole{"Select role"} PickRole -- "Citizen" --> Submit["Submit registration"] PickRole -- "Rescuer" --> RolePwd["Enter organization\npassword"] RolePwd --> Submit AuthChoice -- "Google" --> GoogleBtn["Click Sign in\nwith Google"] AccessOK{{"Access Dashboard\nbased on role"}} --> E1((End)) end subgraph L1_System["AQUAGUARD SYSTEM"] InputLogin --> VerifyPwd["Verify phone number\nand password"] VerifyPwd --> LoginOK{"Valid?"} LoginOK -- "Invalid" --> LoginErr["Show error message"] LoginErr -.-> InputLogin LoginOK -- "Valid" --> CreateToken["Create login session"] Submit --> ValidateReg["Validate information\nand create account"] ValidateReg --> RegOK{"Success?"} RegOK -- "Error" --> RegErr["Show error message"] RegErr -.-> InputReg RegOK -- "OK" --> CreateToken GoogleBtn --> GoogleAuth["Authenticate\nvia Google"] GoogleAuth --> IsNew{"New user?"} IsNew -- "Yes" --> ShowModal["Show role selection\ndialog: Citizen\nor Rescuer"] ShowModal --> SaveRole["Save user role"] SaveRole --> CreateToken IsNew -- "No" --> LoadRole["Load saved role"] LoadRole --> CreateToken CreateToken --> Redirect["Redirect based on role"] Redirect -- "Citizen" --> DashCitizen["Citizen Dashboard"] Redirect -- "Rescuer" --> DashRescuer["Rescue Missions\nPage"] Redirect -- "Admin" --> DashAdmin["Admin Dashboard"] DashCitizen --> AccessOK DashRescuer --> AccessOK DashAdmin --> AccessOK end end

Process Description

  • User Visits the AquaGuard website and chooses a login method: Phone number or Sign in with Google.
  • Phone — Login: Enters phone number and password. System verifies the credentials — shows an error if incorrect, or creates a session if valid.
  • Phone — Register: Fills in personal information (full name, phone, password, gender, date of birth). Then selects a role: Citizen (no extra step) or Rescuer (must enter an organization password issued by the rescue authority).
  • Google: Clicks "Sign in with Google" — the system authenticates via the Google account. First-time users are shown a dialog to choose their role (Citizen or Rescuer) before proceeding.
  • System After successful authentication, automatically redirects the user based on their role: Citizen → Dashboard, Rescuer → Missions page, Admin → Admin Dashboard.
Note: The Admin role can only be assigned by another Admin through the user management page — it cannot be self-registered.

Submitting an Emergency SOS Request

The process by which a citizen sends an emergency rescue request during a flood crisis
flowchart TB subgraph Pool2["PROCESS 2: SUBMIT SOS"] direction TB subgraph L2_Citizen["CITIZEN"] S2((Emergency\nsituation)) --> OpenPage["Open SOS page"] OpenPage --> FillForm["Fill in rescue request:\n- Current location\n- Describe the situation\n- Select urgency level\n- Upload photos"] FillForm --> SendSOS["Press Send SOS"] TrackReq{{"Track request status\nin My Requests"}} --> CheckStatus{"Status?"} CheckStatus -- "In Progress" --> ViewMap["View live tracking map\nwith rescuer location"] CheckStatus -- "Completed" --> Safe{{"Rescued successfully"}} Safe --> E2((End)) end subgraph L2_System["AQUAGUARD SYSTEM"] SendSOS --> GetGPS["Capture GPS coordinates\nfrom device"] GetGPS --> UploadImg["Upload photos to\ncloud storage"] UploadImg --> SaveRequest["Save rescue request\nto database\nStatus: Pending"] SaveRequest --> ShowInQueue["Display request in\nRescue Queue"] ShowInQueue -.-> TrackReq AcceptedNotify["Update status:\nIn Progress"] -.-> CheckStatus CompletedNotify["Update status:\nCompleted"] -.-> CheckStatus end end

Process Description

  • Citizen During a flood emergency, opens the SOS page on the web application.
  • Fills out the rescue request form including: current location (can be auto-detected via GPS), situation description, urgency level (Low / Medium / High / Critical), and photos of the scene (up to 5 images).
  • System Automatically captures GPS coordinates from the device, uploads photos to cloud storage, and saves the request to the database with a "Pending" status.
  • The SOS request appears in the Rescue Queue visible to all available rescuers.
  • Citizen Can track the request status in "My Requests". When a rescuer accepts, the status changes to "In Progress" and the citizen can view the rescuer's live location on the tracking map.
Note: The system automatically refreshes the request list every 10 seconds so the citizen always sees the latest status.

Rescue Request Handling & Operations

Rescuers accept SOS requests, navigate to scene, and complete the rescue
flowchart TB subgraph POOL3["PROCESS 3: HANDLE RESCUE"] direction TB subgraph RESCUER3["RESCUER"] S3((Request)) --> View[View pending requests] View --> Filter[Filter by urgency, age, city] Filter --> Pick[Select a request] Pick --> AM{Accept as?} AM -- Individual --> Acc[Accept individually] AM -- Team --> AccG[Accept as team] Acc --> Go[Navigate to scene] AccG --> Go Go --> Rescue[Perform rescue] Rescue --> OC{Outcome?} OC -- Done --> Mark[Confirm completion] OC -- Cannot reach --> Release[Release back to queue] end subgraph SYS3["AQUAGUARD SYSTEM"] Acc --> SIP[Status: In Progress] AccG --> SIP SIP --> Track3[Start live tracking] Mark --> SC[Status: Resolved] SC --> E3((End)) Release --> SR[Revert to Pending] SR -.-> View AdminAssigned[Status: Assigned] --> NeedAccept[Rescuer must accept to start] NeedAccept -.-> View end subgraph ADMIN3["ADMINISTRATOR"] AView[View all SOS] --> AA{Action?} AA -- Assign rescuer --> AAssign[Pick a rescuer from list] AAssign --> AdminAssigned AA -- Force complete --> AComp[Complete in-progress request] AComp --> SC end end

Process Description

  • Rescuer Views pending SOS list. Filters by urgency, age, gender, city. Accepts individually or as a team.
  • System Updates to "In Progress", records GPS, starts real-time tracking between citizen and rescuer.
  • Rescuer Navigates to scene. On completion → status becomes "Resolved". If unable → "Release" reverts to Pending.
  • Admin Can assign a rescuer → status becomes "Assigned" (rescuer still needs to accept). Can also force-complete any in-progress request.
Status flow: Pending → (Admin assigns) → Assigned → (Rescuer accepts) → In Progress → Resolved. Or: Pending → (Rescuer accepts directly) → In Progress → Resolved.

Flood Zone Management

The process by which administrators update flood zone information on the map, which syncs in real time to all users
flowchart TB subgraph Pool4["PROCESS 4: MANAGE FLOOD ZONES"] direction TB subgraph L4_Admin["ADMINISTRATOR"] S4((Flood\nsituation\nchange)) --> OpenEditor["Open Flood Map\nEditor tool"] OpenEditor --> Action{"Action?"} Action -- "Add new zone" --> ClickMap["Click a location\non the map"] ClickMap --> FillInfo["Enter zone details:\n- Zone name\n- Severity level\n- Water level in meters"] FillInfo --> SaveNew["Save new zone"] Action -- "Edit zone" --> ClickPin["Click an existing\nzone marker"] ClickPin --> EditInfo["Update name,\nseverity, or water level"] EditInfo --> SaveEdit["Save changes"] Action -- "Delete zone" --> ClickDel["Click the zone\nto remove"] ClickDel --> Confirm{"Confirm deletion?"} Confirm -- "Yes" --> Delete["Delete zone marker"] end subgraph L4_System["AQUAGUARD SYSTEM"] SaveNew --> SyncDB["Save to database"] SaveEdit --> SyncDB Delete --> SyncDB SyncDB --> RealtimeSync["Automatically sync\nin real time\nto all users"] end subgraph L4_Users["ALL USERS"] RealtimeSync --> MapUpdate{{"Map updates instantly:\nPurple = Critical\nRed = Severe\nAmber = Moderate\nGreen = Safe"}} MapUpdate --> E4((Users see\nupdated map)) end end

Process Description

  • Administrator Opens the Flood Map Editor tool from the Admin Dashboard.
  • Can perform three actions: Add a new zone (click on the map), Edit an existing zone (click on a marker), or Delete a zone.
  • When adding or editing, the admin enters: Zone name, Severity level (Critical / Severe / Moderate / Safe), and Water level in meters.
  • System Saves the changes to the database and automatically syncs in real time to every user currently viewing the map — no page reload needed.
  • All Users See the map update instantly with color-coded markers: Purple (Critical), Red (Severe), Amber (Moderate), Green (Safe).
Note: The map editor also displays weather overlay layers (rain, wind, clouds, temperature) from a weather service to help the administrator assess conditions before updating zones.

Rescue Team Management

The process of creating rescue teams, inviting members, and coordinating group-based mission acceptance
flowchart TB subgraph Pool5["PROCESS 5: MANAGE TEAM"] direction TB subgraph L5_Leader["TEAM LEADER"] S5((Create\nteam)) --> CreateTeam["Create a new rescue team:\nEnter team name\nand description"] CreateTeam --> InviteStep["Invite a rescuer\nby phone number"] end subgraph L5_System["AQUAGUARD SYSTEM"] CreateTeam --> SysCreate["Create team\nin database\nAssign role: Leader"] InviteStep --> SysValidate["Validate:\n- Phone number exists?\n- Is a rescuer?\n- Not in another team?"] SysValidate --> ValidOK{"Valid?"} ValidOK -- "Error" --> InvErr["Show error message"] ValidOK -- "OK" --> SendInvite["Send invitation\nto the rescuer"] AcceptReq --> SysJoin["Add to team\nas Member\nCancel other invitations"] DeclineReq --> SysDecline["Mark invitation\nas declined"] end subgraph L5_Invitee["INVITED RESCUER"] SendInvite --> SeeInvite["See pending invitation\non the Team page"] SeeInvite --> InvChoice{"Accept?"} InvChoice -- "Accept" --> AcceptReq["Join the team"] InvChoice -- "Decline" --> DeclineReq["Decline invitation"] SysJoin --> TeamReady{{"Now a team member\nCan accept missions\nas a group"}} TeamReady --> E5((End)) end end

Process Description

  • Team Leader Creates a new rescue team by entering the team name and description. The system automatically assigns the creator as Team Leader.
  • The leader invites other rescuers by entering their phone number.
  • System Validates the invitation: the phone number must exist, the person must be a registered rescuer, and they must not already belong to another active team.
  • Invited Rescuer Sees the pending invitation on their "Team" page and can Accept or Decline.
  • Upon accepting, the rescuer becomes a team member and can accept SOS missions as a group rather than individually.
Note: Each rescuer can only belong to one team at a time. Accepting an invitation from one team automatically cancels all other pending invitations.

Family Safety Tracking

The process of adding family members, sharing locations, and monitoring safety status on the live map
flowchart TB subgraph Pool6["PROCESS 6: FAMILY SAFETY"] direction TB subgraph L6_User["USER"] S6((Start)) --> OpenSettings["Open Settings page\nFamily section"] OpenSettings --> AddMember["Add family member:\nEnter phone number\nand relationship"] AddMember --> ViewMap["Open the map\nand enable Family mode"] ViewMap --> SeePins{{"See family members\nlocations on the map"}} SeePins --> WantRoute{"Want directions?"} WantRoute -- "Yes" --> GetRoute["Tap on a\nfamily member"] GetRoute --> SeeRoute{{"View driving route,\ndistance in km,\nand estimated time"}} WantRoute -- "No" --> E6((End)) SeeRoute --> E6 end subgraph L6_System["AQUAGUARD SYSTEM"] AddMember --> SysSave["Save family member\ninformation"] ViewMap --> SysLocate["Get current location\nfrom device GPS\nand sync to server"] SysLocate --> SysFetch["Fetch family members\nlocations from database"] SysFetch --> ShowPins["Display member pins\non the map with\nsafety status colors"] ShowPins -.-> SeePins GetRoute --> CalcRoute["Calculate driving route\nvia mapping service"] CalcRoute -.-> SeeRoute end end

Process Description

  • User Opens the Settings page → Family section to add family members by entering their phone number and relationship.
  • When opening the map and enabling "Family" mode, the system displays family members' locations as pins with colors representing their safety status (Green = Safe, Red = Danger, Orange = Injured, Gray = Unknown).
  • System Automatically captures the user's GPS location and syncs it to the server. Simultaneously fetches family members' locations from the database for display on the map.
  • User Can tap on a family member's pin to get driving directions, including the route, distance in km, and estimated travel time in minutes.
Note: Locations are automatically updated every minute and whenever the user moves more than 50 meters.
