1. Kiến trúc & kỹ thuật1. Architecture & engineering

Cả hai — dùng đúng chỗ.

  • WebSocket (native ws, chung HTTP server trong backend/index.js) cho luồng GPS độ trễ thấp: (a) presence luôn-bật — mỗi client đăng nhập mở 1 socket, gửi presence_location; (b) phòng theo dõi từng ca — citizen ↔ rescuer trao đổi vị trí trực tiếp khi đang cứu hộ.
  • Polling cho dữ liệu ít gấp hơn, nơi đơn giản & chịu lỗi quan trọng hơn realtime: bản đồ lũ poll /api/sos/all mỗi 5s và /api/locations/live mỗi 4s.

Lý do: WebSocket cho phép đẩy vị trí mượt (sub-giây) mà không spam HTTP; còn polling đơn giản, không cần giữ kết nối, hồi phục dễ khi mạng chập chờn — phù hợp đặc thù vùng lũ. Đây là kiến trúc “đẩy cho cái cần tức thời, kéo cho cái cần ổn định”.

Both — each where it fits.

  • WebSocket (native ws, sharing the HTTP server in backend/index.js) for low-latency GPS flows: (a) always-on presence — every logged-in client opens one socket and sends presence_location; (b) per-request tracking rooms — citizen ↔ rescuer exchange location directly during an active rescue.
  • Polling for less urgent data, where simplicity & fault tolerance matter more than realtime: the flood map polls /api/sos/all every 5s and /api/locations/live every 4s.

Why: WebSocket pushes smooth (sub-second) location updates without spamming HTTP; polling is simple, doesn't need a held connection, and recovers easily on flaky networks — a good fit for flood zones. It's a "push what needs to be instant, pull what needs to be stable" architecture.

Trong phòng theo dõi WebSocket, vị trí được broadcast gần như tức thời (mỗi fix, <1s) giữa citizen và rescuer.

Với bản đồ tổng quan, độ trễ là chu kỳ polling: 4–5 giây. Presence trên Redis có TTL 60s — quá 60s không cập nhật thì coi là “offline”. Server còn throttle ghi Redis (di chuyển ≥ ~5,5m hoặc ≥ 2s) để tiết kiệm quota Upstash.

Mục tiêu thực tế: ≤ 5 giây cho bản đồ, < 1 giây cho ca đang theo dõi — đủ cho điều phối cứu hộ (xuồng/người di chuyển chậm).

In a WebSocket tracking room, location is broadcast almost instantly (each fix, <1s) between citizen and rescuer.

For the overview map, the lag is the polling cycle: 4–5 seconds. Redis presence has a 60s TTL — no update past 60s and a user is treated as "offline." The server also throttles Redis writes (move ≥ ~5.5m or ≥ 2s elapsed) to save Upstash quota.

Practical target: ≤ 5 seconds for the map, < 1 second for an active tracking session — enough for coordinating rescue (boats/people move slowly).

Hiện tại:

  • Redis hot store tách toàn bộ tải vị trí khỏi PostgreSQL — presence không bao giờ chạm DB; chỉ fix đầu/cuối của 1 ca theo dõi mới ghi Postgres.
  • Throttle ghi (5,5m / 2s) giảm mạnh số lệnh ghi.
  • Rate limiter (middleware/rateLimit.js) chặn lạm dụng các endpoint nhạy (login, register, gửi thông báo).
  • Index DB đầy đủ cho các truy vấn nóng (status, pending, created_at...).

Giới hạn cần nói thật: backend hiện chạy 1 instance Render free tier. Để chịu hàng nghìn user đồng thời cần (roadmap): nâng plan, chạy nhiều instance sau load balancer, và Redis pub/sub để đồng bộ WebSocket giữa các instance.

Today:

  • Redis hot store takes all location load off PostgreSQL — presence never touches the DB; only the first/last fix of a tracking session is written to Postgres.
  • Write throttling (5.5m / 2s) sharply cuts the number of writes.
  • Rate limiter (middleware/rateLimit.js) blocks abuse of sensitive endpoints (login, register, sending notifications).
  • Full DB indexes for hot queries (status, pending, created_at...).

Honest limit: the backend currently runs as 1 Render free-tier instance. Handling thousands of concurrent users needs (roadmap): a paid plan, multiple instances behind a load balancer, and Redis pub/sub to sync WebSocket state across instances.

Push từ thiết bị qua WebSocket. Frontend dùng navigator.geolocation.watchPosition (enableHighAccuracy, maximumAge: 3000) và gửi presence_location khi vị trí đổi. Server áp throttle: chỉ ghi Redis khi di chuyển ≥ ~5,5m hoặc ≥ 2 giây trôi qua, nên tần suất hiệu dụng ~0,5 Hz khi đứng yên và cao hơn khi di chuyển.

Push from the device over WebSocket. The frontend uses navigator.geolocation.watchPosition (enableHighAccuracy, maximumAge: 3000) and sends presence_location whenever the position changes. The server throttles writes: it only writes to Redis when movement ≥ ~5.5m or ≥ 2 seconds have elapsed, so the effective rate is ~0.5 Hz when stationary and higher while moving.

Thành thật: bản web hiện tại cần kết nối để gửi SOS (POST /api/sos) và stream GPS. Khi mất mạng, WebSocket đóng và frontend tự kết nối lại khi có mạng.

Roadmap (đã thiết kế hướng đi): hàng đợi SOS lưu local (localStorage/IndexedDB hoặc PWA Service Worker + Background Sync) → tự flush khi có tín hiệu; fallback SMS qua Twilio. Xem chi tiết ở Phần 3 (Kịch bản 1) và Phần 5.

Honestly: the current web app needs a connection to send an SOS (POST /api/sos) and stream GPS. When the network drops, the WebSocket closes and the frontend automatically reconnects once a connection returns.

Roadmap (direction already designed): a local SOS queue (localStorage/IndexedDB or PWA Service Worker + Background Sync) → auto-flush when a signal returns; SMS fallback via Twilio. See details in Part 3 (Scenario 1) and Part 5.

Hiện tại: scale dọc — 1 instance Express trên Render, nâng tài nguyên khi cần. Đây là lựa chọn hợp lý cho giai đoạn MVP free-tier và vì WebSocket giữ trạng thái phòng (trackingRooms) trong bộ nhớ tiến trình.

Hướng scale ngang (roadmap): tầng dữ liệu đã sẵn sàng — PostgreSQL (Neon) và Redis (Upstash) đều là dịch vụ độc lập, dùng chung qua 1 client (db.js, redisClient.js). Để scale ngang chỉ cần chuyển trackingRooms sang Redis pub/sub và đặt nhiều instance sau load balancer. App backend gần như stateless ngoài map phòng đó.

Today: vertical — 1 Express instance on Render, scaled up on resources as needed. That's the sensible choice for a free-tier MVP stage and because the WebSocket layer keeps room state (trackingRooms) in the process's memory.

Horizontal direction (roadmap): the data layer is already ready for it — PostgreSQL (Neon) and Redis (Upstash) are both independent services, accessed through a single shared client (db.js, redisClient.js). Scaling horizontally would just need moving trackingRooms to Redis pub/sub and placing multiple instances behind a load balancer. The backend app is nearly stateless outside that one room map.

2. Dữ liệu & bản đồ lũ2. Data & flood map

Bản đồ AquaGuard chồng nhiều lớp:

  • Vùng lũ (flood zones): do admin nhập/biên tập (AdminFloodMapEditor.jsx), lưu ở Firebase Firestore (collection flood_zones) với mức độ (critical/severe/moderate/low/safe) và mực nước. Đây là dữ liệu người vận hành kiểm duyệt, không phải tự động.
  • Thời tiết thực: lớp phủ Windy API (mưa, gió, mây, nhiệt độ, áp suất, sóng) — dữ liệu khí tượng thật.
  • SOS & vị trí cứu hộ: báo cáo người dùng thật-thời gian-thực (citizen gửi SOS, rescuer stream vị trí qua Redis).

Roadmap: tích hợp API cảnh báo của KTTV/chính quyền và cảm biến IoT mực nước (menu admin đã có mục “Flood Sensors”).

The AquaGuard map layers several sources:

  • Flood zones: entered/edited by an admin (AdminFloodMapEditor.jsx), stored in Firebase Firestore (flood_zones collection) with a severity level (critical/severe/moderate/low/safe) and water level. This is human-moderated data, not automatic.
  • Real weather: a Windy API overlay (rain, wind, clouds, temperature, pressure, waves) — real meteorological data.
  • SOS & rescuer location: genuine real-time user reports (citizens send SOS, rescuers stream location via Redis).

Roadmap: integrate the national hydro-meteorological agency's/government's alert API and IoT water-level sensors (the admin menu already has a "Flood Sensors" entry).

  • Vùng lũ: độ chính xác phụ thuộc người admin cập nhật — đây là dữ liệu được kiểm duyệt, đánh đổi “tự động real-time” lấy “đáng tin cậy”.
  • Vị trí người/cứu hộ: phụ thuộc GPS thiết bị (enableHighAccuracy). Geohash của Redis GEO có sai số ~0,6m nên hệ thống cố ý đọc lat/lng gốc từ hash thay vì toạ độ GEO đã giải mã, để tránh sai lệch.
  • Thời tiết: độ phân giải theo Windy.

Roadmap: điểm tin cậy (confidence score) phân biệt dữ liệu cảm biến đã xác minh vs báo cáo người dùng — xem Phần 4, Stress-test B.

  • Flood zones: accuracy depends on the admin who updates it — this is moderated data, trading "automatic real-time" for "trustworthy."
  • People/rescuer location: depends on device GPS (enableHighAccuracy). Redis GEO's geohash has ~0.6m of error, so the system deliberately reads the original lat/lng from the hash instead of the decoded GEO coordinate, to avoid drift.
  • Weather: resolution follows Windy's.

Roadmap: a confidence score distinguishing verified sensor data from user reports — see Part 4, Stress-test B.

Ưu tiên GPS thực của nạn nhân để điều phối cứu hộ — vì đó là vị trí con người cần đến. Bản đồ lũ là lớp ngữ cảnh (đường nào ngập, vùng nào nguy hiểm) giúp chọn lối tiếp cận, không thay thế vị trí nạn nhân.

Trong code, request pending/in_progress luôn lấy toạ độ live (COALESCE(loc.latitude, r.latitude) + overlay Redis) làm nguồn sự thật về vị trí nạn nhân.

Roadmap: khi GPS rơi vào vùng bất khả thi (giữa sông), đối chiếu với lớp địa hình/lũ để gắn cờ “GPS nghi sai” — xem Phần 3, Kịch bản 5.

The victim's actual GPS wins for dispatching rescue — because that's the location a human needs to reach. The flood map is a context layer (which roads are flooded, which areas are dangerous) that helps pick an approach route, not a replacement for the victim's location.

In code, a pending/in_progress request always reads the live coordinate (COALESCE(loc.latitude, r.latitude) plus a Redis overlay) as the source of truth for the victim's location.

Roadmap: when GPS lands somewhere impossible (mid-river), cross-check against the terrain/flood layer and flag "GPS possibly wrong" — see Part 3, Scenario 5.

Các lớp động (SOS, vị trí cứu hộ) cập nhật mỗi 4–5 giây qua polling, đủ nhanh cho điều phối. Lớp vùng lũ cập nhật khi admin chỉnh — đây là điểm yếu với lũ quét đột ngột vì phụ thuộc con người.

Bù lại cho lũ quét: chính các SOS thời gian thực và lớp mưa Windy đóng vai trò tín hiệu sớm. Roadmap: cảm biến IoT mực nước + cảnh báo tự động để vùng lũ tự cập nhật giây/phút.

The dynamic layers (SOS, rescuer location) update every 4–5 seconds via polling — fast enough for dispatch. The flood zone layer updates only when an admin edits it — a weak point for sudden flash floods, since it depends on a human.

What compensates for flash floods: real-time SOS reports and the Windy rain layer act as early signals. Roadmap: IoT water-level sensors + automatic alerts so flood zones update themselves within seconds/minutes.

Đã lưu lịch sử vận hành: rescue_requests (kèm toạ độ, thời điểm, mức độ), rescue_request_logs (toàn bộ thay đổi trạng thái — audit trail), user_locations (vị trí đầu/cuối ca), audit_logs. Đây là kho dữ liệu vàng cho phân tích sau lũ.

Roadmap: dùng lịch sử SOS theo toạ độ để tạo “bản đồ nhiệt vùng nguy cơ” và huấn luyện mô hình dự báo.

Operational history already stored: rescue_requests (with coordinates, timing, severity), rescue_request_logs (every status change — an audit trail), user_locations (first/last position per session), audit_logs. This is a goldmine for post-flood analysis.

Already used: the admin analytics pages (/api/analytics/overview|users|rescue) report by area, time, and team performance.

4. An toàn, bảo mật & riêng tư4. Safety, security & privacy

Truy cập bị giới hạn theo vai trò (RBAC) + JWT bắt buộc:

  • Người thân chỉ thấy vị trí thành viên gia đình đã accepted trong family_connections (liên kết hai chiều, có đồng ý).
  • Rescuer thấy SOS của nhóm mình được phân (/api/sos/team); admin thấy toàn cảnh để điều phối.
  • Vị trí live có TTL 60s trên Redis — hết khẩn cấp là tự biến mất, không lưu vết liên tục.

Mỗi yêu cầu phải có token hợp lệ; WebSocket cũng xác thực JWT khi bắt tay.

Khi một đội nhả ca, ca về pending và tracking_cancelled phát đi. Roadmap: thông báo tường minh “rescue bị trễ / đội mới đang tới” để trấn an — quản lý lo lắng của nạn nhân là vấn đề UX thật.

Chưa. Hiện admin có thể phân lại bất kỳ nhóm nào (kể cả nhóm chuyên dụng nếu được tạo). Roadmap: cờ “không có lối — cần đường không” nổi trên dashboard chỉ huy để kích hoạt đơn vị chuyên dụng/trực thăng.

Kịch bản 4 — “Nạn nhân không có điện thoại”

Một cụ bà 80 tuổi mắc kẹt trên mái nhà, không có smartphone. Con gái ở Hà Nội thấy tin lũ và biết đại khái mẹ ở đâu.

Thật: đây là tính năng proxy-SOS — chưa có, và là ưu tiên cao nhất trong roadmap người dùng. Cần thiết cho người già/nông thôn không smartphone. Thiết kế: người thân thả ghim + nhập địa chỉ gần đúng → tạo SOS thay mặt, đánh dấu “gửi hộ” + bán kính bất định. Hạ tầng SOS (lưu toạ độ, ảnh, mô tả, phân công) đã sẵn để tái dùng.

Schema rescue_requests đã có trường location (text mô tả) tách khỏi latitude/longitude, nên một ca chỉ có địa chỉ mô tả vẫn lưu được. Roadmap: vẽ vòng tròn bán kính bất định quanh điểm phỏng đoán trên bản đồ để cứu hộ biết phạm vi cần tìm, thay vì một điểm giả-chính-xác.

Cơ chế này đã hoạt động cho người thân liên kết: chuỗi trạng thái push rõ ràng — pending → đã nhận (“Đội cứu hộ đang đến”, kèm tên) → theo dõi GPS realtime → hoàn tất (“đã an toàn”), tất cả qua in-app + email. Khi proxy-SOS hoàn thiện (câu 13), người gửi hộ kế thừa đúng luồng theo dõi này.

Cần khi có proxy-SOS — hiện chưa có dedupe. Roadmap: phát hiện trùng theo lân cận toạ độ + cùng người được cứu (đối chiếu danh tính/SĐT người nhà) → gộp thành một ca và liên kết nhiều người báo, tránh điều hai đội cho một nạn nhân. Logic gom cụm dùng lại được Redis GEO.

Kịch bản 5 — “GPS sai vị trí”

Một nạn nhân gửi SOS. GPS đặt họ giữa sông — rõ ràng bất khả thi. Đội điều tới đó không thấy ai.

Chưa có kiểm tra tự động. Nền tảng đã có: lớp vùng lũ + dữ liệu địa hình có thể đối chiếu. Roadmap: khi toạ độ rơi vào vùng nước/bất khả thi → gắn cờ “GPS nghi sai” cho cứu hộ và đề nghị nạn nhân chỉnh ghim. Cross-check GPS với lớp bản đồ lũ + địa hình.

Hiện app dùng Geolocation API của trình duyệt — vốn đã tự kết hợp GPS + Wi-Fi + cell tower ở tầng hệ điều hành. Khi gửi SOS, nạn nhân nhập kèm địa chỉ mô tả (trường location) làm phương án con người. Roadmap: chuỗi fallback tường minh — GPS → Wi-Fi/cell → ghim tay trên bản đồ, hiển thị rõ đang dùng nguồn nào.

Chưa, nhưng đơn giản và đáng làm. Hiện SOS lấy toạ độ tự động từ GPS. Roadmap: thêm bước “kéo ghim để chỉnh vị trí” khi gửi SOS — một override thủ công đơn giản vẫn tốt hơn không có lựa chọn nào. Trường lat/lng đã sẵn để nhận giá trị chỉnh tay.

Cơ chế đã có: đội bấm “Nhả ca” → ca về pending, ghi last_cancelled_by/last_cancelled_at và một dòng trong rescue_request_logs với ghi chú lý do. Đây là audit trung lập (ghi nhận sự việc, không phạt đội), và ca lập tức sẵn sàng cho đội khác/admin nhận lại — đảm bảo liên tục vận hành sau một lần điều hụt.

PHẦN 4 — Câu hỏi stress-test

Nhiều sự cố xảy ra đồng thời — câu hỏi khó nhất, bộc lộ độ bền thật của hệ thống.

Stress-test A — “Mọi thứ hỏng cùng lúc”

3h sáng, đỉnh lũ. Trạm sóng sập, mất điện 3 huyện, 200 SOS đã bắn. Server trung tâm 98% CPU. Hai trưởng đội báo app crash.

Trung thực về hiện trạng: backend là 1 instance Render → đây là điểm-chết-đơn. Cái đã có giúp giảm tải: Redis gánh toàn bộ vị trí (không đụng DB), throttle ghi, rate limiting, index DB, và Redis suy biến mượt.

Định hướng ba lớp (xem câu trả lời mẫu đầy đủ ở Phần 5): (1) hàng đợi SOS phía thiết bị + retry; (2) message broker hấp thụ đỉnh; (3) load balancer tự failover sang instance dự phòng khi CPU vượt ngưỡng. Lưu ý: ở bản hiện tại các lớp 2–3 là roadmap, không nên nói là đã có khi demo.

Vẫn được giữ. Phân công lưu bền ở PostgreSQL (status='in_progress', assigned_to, assigned_group_id) — không phụ thuộc app còn chạy hay socket còn mở. Rescuer mở lại app, đăng nhập (phiên JWT khôi phục), ca vẫn của họ và họ tiếp tục theo dõi. Ca chỉ về pending khi chủ động nhả — không tự rớt vì crash.

Hiện tại: phiên đăng nhập được lưu (khôi phục offline), và AuthContext có cơ chế fallback dữ liệu Firebase khi backend không truy cập được. Tile Google Maps đã xem được trình duyệt cache một phần. Thật: danh sách SOS và phân công chưa cache offline (đang là dữ liệu poll). Roadmap: PWA cache tile + cache “ca đang nhận của tôi” + danh sách nạn nhân gần nhất để dùng khi mất server.

Hiện có: endpoint /api/health + UptimeRobot ping mỗi 5’ (cảnh báo khi backend chết/ngủ). Log server (Render) theo dõi lỗi Redis/DB. Thật: chưa có cảnh báo theo ngưỡng tải/độ trễ và chưa có quy trình on-call chính thức. Roadmap: dashboard sức khoẻ thời gian thực + cảnh báo on-call + đường dây nóng/leo thang thủ công — phải có con người dự phòng, không chỉ tự động.

Stress-test B — “Dữ liệu xấu làm hỏng bản đồ”

Mười tình nguyện viên gửi báo cáo lũ giả. Bản đồ giờ hiện vùng ngập nơi thực ra khô, và khô nơi thực ra ngập. Đội bị định tuyến sai.

Hiện tại — phòng vệ bằng kiểm soát nguồn: vùng lũ không mở cho người dùng tự đăng; chỉ admin biên tập (AdminFloodMapEditor). Tức là kịch bản “10 tình nguyện viên bơm dữ liệu giả” bị chặn từ gốc — không có kênh crowdsource công khai để đầu độc.

Roadmap (khi mở crowdsource để cập nhật nhanh hơn): ngưỡng tối thiểu số báo cáo, đối chiếu chéo với cảm biến, kiểm duyệt của admin trước khi công khai.

Vì vùng lũ do admin quản trong Firestore, admin sửa/xoá điểm ngay và mọi client thấy thay đổi ở chu kỳ poll kế tiếp (giây). Không cần deploy lại. Khi demo: mở AdminFloodMapEditor, xoá một điểm sai → nó biến mất trên bản đồ người dùng gần như tức thì.

Chưa. Hiện mọi vùng lũ là dữ liệu admin (mặc nhiên “đã kiểm duyệt”), còn lớp thời tiết là Windy (dữ liệu khí tượng). Roadmap: khi có cảm biến IoT + báo cáo người dùng, hiển thị confidence score và biểu tượng phân biệt “cảm biến xác minh” vs “người dùng báo” — chỉ báo niềm tin trực quan rất quan trọng khi cứu hộ ra quyết định sinh tử.

Stress-test C — “Một ca im lặng 4 giờ”

Một nạn nhân gửi SOS 4 giờ trước. Đội được phân trong 5 phút. Nhưng 4 giờ sau vẫn “đang trên đường”, không cập nhật, không xác nhận. Gia đình gọi tổng đài đòi câu trả lời.

Thật: chưa có auto-escalation. Dữ liệu cần thiết đã sẵn — assigned_at, resolved_at, và toàn bộ rescue_request_logs có timestamp, nên đo “im lặng bao lâu” là khả thi ngay. Roadmap (đáng demo): job định kỳ phát hiện ca in_progress quá ngưỡng (ví dụ 60’) không có cập nhật → tự leo thang lên admin và cảnh báo. Nói rõ con số quy tắc khi trình bày.

Có. Hệ thống theo dõi cả hai phía: SOS lưu rescuer_latitude/longitude, và vị trí live của rescuer nằm trong Redis (/api/locations/live/:userId) với fallback “vị trí cuối được biết” ở user_locations khi họ offline. Admin tra được điểm cuối của đội gán cho ca đó — không chỉ theo dõi nạn nhân.

Chưa tự gắn cờ. Tín hiệu đã có: presence Redis hết hạn sau 60s (biết rescuer mất kết nối) và mốc thời gian log. Roadmap: logic theo dõi-kép — nếu rescuer đang giữ ca mà mất tín hiệu kéo dài, gắn cờ “đội có thể gặp nguy” trên dashboard. Cứu hộ cũng có thể trở thành nạn nhân — đây là điểm cần thể hiện rõ.

PHẦN 5 — Câu trả lời mẫu cho demo: dung lượng server & failover

Trả lời chuẩn bị sẵn cho Stress-test A, câu 1 — dùng khi demo/pitch trực tiếp.

⚠ Lưu ý trung thực quan trọng: kiến trúc ba lớp dưới đây (queue thiết bị · message broker · failover tự động) là câu trả lời định hướng / mục tiêu thiết kế, chưa được triển khai trong bản hiện tại (đang chạy 1 instance Render free-tier). Khi pitch, hãy trình bày nó như kiến trúc mục tiêu và nói rõ phần nào đã có (Redis tách tải, throttle, rate-limit, suy biến mượt, lưu bền Postgres). Đừng tuyên bố các con số tải/ failover là kết quả đo thật nếu chưa chạy load test.

Câu mở đầu — nói trước:

“Câu hỏi rất hay — và đây chính là kịch bản chúng tôi thiết kế để chống chịu. Hệ thống dùng phòng vệ ba lớp: hàng đợi cục bộ trên thiết bị, một message broker hấp thụ đỉnh tải, và failover tự động sang máy dự phòng. Không SOS nào bị bỏ.”

1 · Hàng đợi phía thiết bị — Khi nạn nhân bấm SOS, yêu cầu được ghi vào bộ nhớ cục bộ trước khi gọi server. Nếu server quá tải, thiết bị tự thử lại mỗi 5 giây. Người dùng thấy “Đã gửi SOS” ngay, không phải chờ server xác nhận.

2 · Message broker hấp thụ đỉnh — SOS vào hàng đợi trước khi chạm server chính. Nếu 500 cảnh báo đến trong 60 giây, broker giữ lại và nạp vào server theo tốc độ xử lý được — không bỏ request, chỉ trễ chút. Như một bộ giảm xóc giữa nhu cầu và năng lực.

3 · Failover tự động — Nếu CPU server chính vượt 85%, load balancer tự chuyển traffic mới sang máy dự phòng trong <30 giây. Máy dự phòng luôn chạy đồng bộ nên không mất dữ liệu khi chuyển.

Câu kết — chốt câu trả lời:

“Nguyên tắc cốt lõi: chúng tôi không bao giờ bắt server và nạn nhân phải đồng bộ cùng một thời điểm. Thiết bị xếp hàng, broker đệm, failover hứng nốt phần còn lại. Mục tiêu: 800 SOS đồng thời, 0 bị bỏ.”

Bẫy cần tránh:

  • Nói “nó sẽ không sập” — nghe như chưa kiểm thử. Luôn nêu cơ chế, không chỉ kết quả.
  • Đáp mơ hồ “cloud auto-scaling” — quá chung. Nêu ngưỡng, thời gian phản hồi, công nghệ.
  • Tuyên bố đã load test mà không có số — giám khảo sẽ hỏi đỉnh tải & kết quả.
  • Quên phương án con người — nếu MỌI lớp tự động hỏng, phải có quy trình thủ công.
  • Chỉ bảo vệ server — nửa câu trả lời nằm ở thiết bị (queue cục bộ, retry).

Những gì đã thật sự có trong AquaGuard hôm nay (nên nhấn mạnh để giữ uy tín): Redis hot store tách 100% tải vị trí khỏi Postgres; throttle ghi (5,5m/2s); rate limiting theo endpoint; SOS lưu bền Postgres ngay khi gửi (không mất khi client rớt); Redis suy biến mượt (mất Redis → fallback Postgres); phân công bền vững qua app crash; health check + UptimeRobot.
